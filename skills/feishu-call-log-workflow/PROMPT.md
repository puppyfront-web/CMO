# Execution Prompt

这份文档是 `feishu-call-log-workflow` 的内置执行提示词契约。

默认要求：

- 每次执行这条 workflow 时，都应默认遵循这里的规则
- 不需要用户每次重新手动贴这份 prompt
- 如果用户给了额外要求，只能在不破坏这里的固定输出契约前提下附加执行

## Built-In System / Instruction Prompt

```md
你正在执行一个“客户通话录音分析并写入飞书”的固定工作流。

你的目标不是自由发挥，而是严格完成以下流水线：

1. 接收本地录音文件路径
2. 复用当前默认飞书表；只有用户明确要求换表时才替换
3. 如果当前没有默认飞书表：
   - 优先使用用户明确给出的飞书表链接
   - 如果用户没有给出，则自动创建一个新的飞书电子表格，并保存为默认表
4. 调用本地转写能力生成 transcript
5. 从 transcript 中提取固定字段：
   - 日期
   - 客户名
   - 电话
   - 客户类别
   - 需求
   - 对接阶段
   - 备注
6. 生成一份 Mermaid mindmap 内容
7. 创建一篇飞书纯脑图文档
8. 将一行写入飞书电子表格，列顺序必须严格为：
   - 日期
   - 客户名
   - 电话
   - 客户类别
   - 需求
   - 对接阶段
   - 打电话录音脑图
   - 备注

必须遵守以下规则：

- `打电话录音脑图` 列只存飞书文档链接，不放脑图文字
- `备注` 列只放重要补充信息，例如风险、限制、关键顾虑、重要背景、强约束、明确下一步动作
- 如果某字段未提取到，使用 `未提取到`
- 不要把长篇解释写入表格
- 脑图文档正文只保留脑图，不要再写大段分析文字
- 脑图文档标题优先使用：`客户名-行业-日期`
- 如果字段不足，按以下顺序兜底：
  - `客户名-通话脑图-日期`
  - `行业-通话脑图-日期`
  - `通话脑图-日期`

在执行过程中：

- 优先使用 skill 内已有脚本和状态文件
- 把 workflow 当作显式 stages 执行，不要绕过主流程
- 如需新增能力，优先作为 stage 或 extension 挂入，不要破坏现有表结构和输出契约
```

## Optional User Prompt Template

```md
请使用 `feishu-call-log-workflow` 处理这段客户通话录音：

录音文件路径：
`/absolute/path/to/audio.mp3`

执行要求：

1. 如果已有默认飞书表，则直接继续写入
2. 如果我这次明确给了新的飞书表链接，则替换默认表并写入新表
3. 输出固定字段：
   - 日期
   - 客户名
   - 电话
   - 客户类别
   - 需求
   - 对接阶段
   - 备注
4. 生成飞书纯脑图文档
5. 将脑图文档链接写入 `打电话录音脑图` 列
6. 将重要补充信息写入 `备注` 列
7. 完成后返回：
   - 写入到哪张飞书表
   - 新增的是哪一行
   - 脑图文档链接
```

## Field Extraction Guide

建议模型按下面口径提取：

- `日期`
  - 优先使用通话发生日期
  - 如果录音文件名中带日期且正文未明确冲突，可合理推断
- `客户名`
  - 优先取通话中明确称呼
- `电话`
  - 没有出现则填 `未提取到`
- `客户类别`
  - 尽量用“行业 / 身份 / 业务类型”组合表达
- `需求`
  - 只保留核心诉求，合并成 1-5 条
- `对接阶段`
  - 例如：`初步沟通 / 待评估`、`需求确认中`、`待报价`
- `备注`
  - 只记录高价值补充信息
  - 例如：客户对安全很敏感、已有旧系统、预算谨慎、必须本地部署、要和某工具兼容

## When To Add New Nodes

如果未来要扩展这条流水线：

- 必经主流程变化：修改 `stages`
- 可选附加能力：添加到 `extensions`
- 推理型新增能力：优先加 agent node
- 工具型或系统集成能力：优先加 sub-workflow 或 extension
